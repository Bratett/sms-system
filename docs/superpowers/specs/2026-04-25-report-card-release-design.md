# Report Card Release + Acknowledgements — Design

**Date:** 2026-04-25
**Status:** Approved
**Tier:** 2 — final sub-project of item #6 (Parent Portal Upgrade). Sub-project **D2**.

## 1. Context

Sub-projects B (PR #26 — messaging), C (PR #27 — parent-initiated workflows), and D1 (PR #28 — circular acknowledgements) shipped. D2 closes the Parent Portal Upgrade with **report card release + acknowledgements**.

**Today's state:**
- `TerminalResult` rows are visible to parents *immediately upon compute* — no gate. The UI's "No results have been published" empty-state is cosmetic.
- `publishResultsAction` exists but is a no-op stub: writes audit, returns. No flag set, no notification fires, parent access not gated.
- `ReportCardPdfCache` exists per `(studentId, termId)`. Generated by staff via `renderReportCardPdfAction` (gated on `REPORT_CARDS_GENERATE`). Parents cannot download the PDF — there's no parent-facing path.
- The cache is only invalidated on remarks edits; score edits silently leave stale PDFs.
- `RESULT_PUBLISHED` notification event exists in the registry but is never fired.

This design adds: (a) a `ReportCardRelease` per-arm-per-term entity that gates parent visibility of both the live HTML view and the PDF; (b) a `ReportCardAcknowledgement` model mirroring D1's pattern; (c) parent-scoped PDF download; (d) a dedicated `/students/results-release` admin page with stats + chase + re-release; (e) cache invalidation on every mutation that affects scores or remarks.

## 2. Scope

**In scope**
- `ReportCardRelease` model (per `Term × ClassArm`)
- `ReportCardAcknowledgement` model (per release × student × household — twins in same arm get separate ack rows)
- 2 new permissions: `REPORT_CARDS_DOWNLOAD_OWN`, `REPORT_CARDS_RELEASE_TRACK`
- 2 new notification events: `REPORT_CARD_RELEASED`, `REPORT_CARD_REMINDER_SENT`
- Pure targeting helper for "students in arm-term release"
- Server actions: release, re-release (with optional ack reset), stats, details, chase, parent acknowledge, parent PDF URL, parent release lookup, admin queue
- Parent portal: `/parent/results` extended with "not yet released" empty state, Download PDF button, acknowledge button, acknowledged badge
- Admin: new `/students/results-release` page with arm-level table + drawer + chase + CSV
- `publishResultsAction` becomes a thin wrapper that delegates to `releaseReportCardsAction`
- Cache invalidation wired into `saveSubjectMarksAction`, `updateSubjectMarkAction`, `deleteSubjectMarkAction`, `recomputeTerminalResultAction`

**Out of scope**
- Two-step submit-then-approve workflow (class teacher submits, academic master approves)
- Per-term-school-wide release (granular per-arm only)
- Configurable cooldown duration (24-hour fixed, matching D1)
- Configurable notification channels at release time (channels come from `EVENT_CHANNELS` defaults / per-user prefs)
- Auto-reminder schedules (admin-triggered chases only, matching D1)
- Sealed/immutable post-release marks (edits stay free; cache invalidates; staff explicitly re-releases for notify)
- Reuse / generalisation of `CircularAcknowledgement` to a polymorphic acknowledgement table

## 3. Architecture

**Approach: dedicated release sub-module.** Place new code under `src/modules/academics/release/`:
- `release-targeting.ts` — pure helper(s) for "students in arm-term release" + household resolution
- `release-notifications.ts` — fan-out wrapper (release + chase events)
- `actions/release.action.ts` — admin-side release / re-release / stats / details / chase / queue
- `actions/parent-release.action.ts` — parent-scoped acknowledge / PDF URL / release lookup

The existing `publishResultsAction` (no-op stub) becomes a thin delegating wrapper to preserve its route + permission. Don't break existing callers.

The existing `renderReportCardPdfAction` is refactored: extract a non-permissioned `_renderReportCardPdfInternal({ studentId, termId, schoolId })` that does the actual work; the public action is a thin wrapper that adds the `REPORT_CARDS_GENERATE` gate. The new parent-scoped action calls the internal helper after its own (parent-side) eligibility check.

Data invariants ("a release row gates parent visibility") are enforced at the action layer in:
- `getChildResultsAction` (HTML view gate)
- `getMyReportCardPdfUrlAction` (PDF gate)
- `acknowledgeReportCardAction` (acknowledge gate)

Cache freshness is enforced via:
- `invalidateReportCardCacheAction` called from every score / remark mutation
- The existing `isReportCardCacheFresh` check on every render path

## 4. Data Model

### `ReportCardRelease`

```prisma
model ReportCardRelease {
  id                  String   @id @default(cuid())
  schoolId            String
  termId              String
  classArmId          String
  releasedByUserId    String?
  releasedAt          DateTime @default(now())
  lastReminderSentAt  DateTime?
  createdAt           DateTime @default(now())
  updatedAt           DateTime @updatedAt

  school          School      @relation("SchoolReportCardRelease", fields: [schoolId], references: [id])
  term            Term        @relation(fields: [termId], references: [id], onDelete: Cascade)
  classArm        ClassArm    @relation(fields: [classArmId], references: [id], onDelete: Cascade)
  releasedBy      User?       @relation("ReleasedReportCards", fields: [releasedByUserId], references: [id], onDelete: SetNull)
  acknowledgements ReportCardAcknowledgement[]

  @@unique([termId, classArmId])
  @@index([schoolId, termId])
  @@index([classArmId])
  @@index([releasedByUserId])
}
```

**Key choices:**
- `@@unique([termId, classArmId])` — one release per arm per term. Re-release updates `releasedAt` (and optionally clears acks) rather than inserting a duplicate.
- `onDelete: Cascade` on `termId` / `classArmId`.
- `releasedByUserId` nullable + `SetNull` for GDPR-safe staff deletion.
- No `status` enum — release is implicit by row existence.

### `ReportCardAcknowledgement`

```prisma
model ReportCardAcknowledgement {
  id                    String    @id @default(cuid())
  releaseId             String
  studentId             String
  householdId           String
  acknowledgedByUserId  String?
  acknowledgedAt        DateTime  @default(now())
  createdAt             DateTime  @default(now())

  release        ReportCardRelease @relation(fields: [releaseId], references: [id], onDelete: Cascade)
  student        Student           @relation(fields: [studentId], references: [id], onDelete: Cascade)
  household      Household         @relation(fields: [householdId], references: [id], onDelete: Cascade)
  acknowledgedBy User?             @relation("AcknowledgedReportCards", fields: [acknowledgedByUserId], references: [id], onDelete: SetNull)

  @@unique([releaseId, studentId, householdId])
  @@index([releaseId])
  @@index([studentId])
  @@index([householdId])
  @@index([acknowledgedByUserId])
}
```

**Key choices:**
- Unique key includes `studentId` so siblings in the same arm-term get independent ack rows. (Differs from `CircularAcknowledgement` which is one ack per household per circular.)
- `onDelete: Cascade` on all three primary FKs.
- `acknowledgedByUserId: SetNull` for GDPR.

### `TerminalResult` is **not** modified

Release is per-arm-per-term, not per-student. Encoding it on `TerminalResult` would require N rows touched per release event. The `ReportCardRelease` lookup is one indexed query and stays normalized.

### Back-relations

- `Term.reportCardReleases ReportCardRelease[]`
- `ClassArm.reportCardReleases ReportCardRelease[]`
- `School.reportCardReleases ReportCardRelease[] @relation("SchoolReportCardRelease")`
- `User.releasedReportCards ReportCardRelease[] @relation("ReleasedReportCards")`
- `User.acknowledgedReportCards ReportCardAcknowledgement[] @relation("AcknowledgedReportCards")`
- `Student.reportCardAcknowledgements ReportCardAcknowledgement[]`
- `Household.reportCardAcknowledgements ReportCardAcknowledgement[]`

## 5. Release Semantics + Edit Behavior

A release is binary — a `ReportCardRelease` row exists or doesn't. No state enum.

| Event | Effect |
|---|---|
| **Initial release** | Insert `ReportCardRelease` row. Notify targeted parents via `REPORT_CARD_RELEASED`. Audit. |
| **Silent re-release** (score/remark edit, no admin action) | `invalidateReportCardCacheAction` called for affected students. PDF regenerates on next download. No row mutation, no notification. Acks preserved. |
| **Explicit re-release with notification** | `releasedAt` and `lastReminderSentAt` updated. Optionally (admin checkbox) all `ReportCardAcknowledgement` rows for this release deleted in a transaction. Re-fires `REPORT_CARD_RELEASED` (with `isReRelease: true` for body prefix). Audit. |
| **Chase reminder** | 24-hour cooldown via `lastReminderSentAt`. Pending households' guardian users notified via `REPORT_CARD_REMINDER_SENT`. Cooldown stamp updated. Audit. |
| **Acknowledge (parent)** | `ReportCardAcknowledgement` upsert (idempotent on unique key). Audit. |

### Pre-release access semantics

| Surface | No release row | Release row exists |
|---|---|---|
| `/parent/results` HTML | "Term results not yet released" empty state | Live scores + PDF button + acknowledge button |
| `getMyReportCardPdfUrlAction` | `{ error: "Report card not yet released" }` | Signed R2 URL (regenerates if cache stale) |
| Staff results-management | Unaffected | Unaffected |
| `renderReportCardPdfAction` (staff) | Allowed | Allowed |

### Pre-release "ready" guard

Admin release page shows per-arm `students_with_results / students_enrolled`. Releasing with incomplete coverage shows a confirmation dialog: "N students will be excluded from this release; they'll be auto-included when their results are computed. Continue?" Admin confirms or cancels. Release row is created either way; missing students gain access automatically when their results land later.

### Idempotency

- Initial release on already-released arm → `{ error: "Already released. Use re-release if you need to refresh." }`
- Acknowledge on unique constraint → `{ success: true }` (idempotent)
- Chase respects 24-hour cooldown
- Re-release works whether or not acks exist; "Reset acknowledgements" is the explicit toggle

## 6. Permissions

Two new constants in `src/lib/permissions.ts`:

```ts
  REPORT_CARDS_DOWNLOAD_OWN: "academics:report-cards:download-own",
  REPORT_CARDS_RELEASE_TRACK: "academics:report-cards:release-track",
```

Grants:

| Permission | Granted to |
|---|---|
| `REPORT_CARDS_DOWNLOAD_OWN` | `parent` |
| `REPORT_CARDS_RELEASE_TRACK` | `headmaster`, `assistant_headmaster_academic`, `academic_master` (mirror `RESULTS_PUBLISH` set) |

`super_admin` inherits via `ALL_PERMISSIONS`.

**Reuse of existing permissions:**
- **Releasing** uses existing `RESULTS_PUBLISH` (already granted to academic admins; the stubbed `publishResultsAction` already gates on it).
- **HTML view of results** uses existing `RESULTS_READ` (parents already hold it). Release row is the gate.
- **Staff PDF render** uses existing `REPORT_CARDS_GENERATE`.

**Eligibility vs permission.** `REPORT_CARDS_DOWNLOAD_OWN` only permits the action; the per-call rule "caller is guardian of student AND release row exists" is enforced inside `getMyReportCardPdfUrlAction`. Non-guardian / pre-release attempts return `{ error: "Report card not found" }` (generic, no leak).

## 7. Server Actions

### `src/modules/academics/release/actions/release.action.ts`

| Action | Permission | Purpose |
|---|---|---|
| `releaseReportCardsAction({ termId, classArmId })` | `RESULTS_PUBLISH` | Validates arm + term in school; rejects double-release. Inserts `ReportCardRelease`. Resolves targeted households. Fans out `REPORT_CARD_RELEASED`. Audit. Returns `{ data: { releaseId } }`. |
| `reReleaseReportCardsAction({ releaseId, resetAcknowledgements })` | `RESULTS_PUBLISH` | Updates `releasedAt`. Clears `lastReminderSentAt`. If `resetAcknowledgements`, deletes ack rows in tx. Re-fires `REPORT_CARD_RELEASED` with `isReRelease: true`. Audit. |
| `getReleaseStatsAction(releaseId)` | `REPORT_CARDS_RELEASE_TRACK` | Returns `{ targetedStudents, acknowledgedStudents, pendingStudents, lastReminderSentAt, canSendReminder, releasedAt, releasedBy }`. Targeted = students with active enrollment in arm at query time. |
| `getReleaseDetailsAction(releaseId)` | `REPORT_CARDS_RELEASE_TRACK` | Per-student rows: `{ studentId, studentName, householdId, householdName, acknowledged, acknowledgedAt, acknowledgedBy }`. Pending-first sort. |
| `chaseReleaseAction(releaseId)` | `REPORT_CARDS_RELEASE_TRACK` | 24h cooldown. Zero-pending rejection. Fan-out to pending households' guardian users. Updates `lastReminderSentAt`. Audit. |
| `getReleaseQueueAction({ termId? })` | `REPORT_CARDS_RELEASE_TRACK` | Admin page table: `[{ classArmId, classArmName, programmeName, studentsEnrolled, studentsWithResults, releaseRow, ackProgress }]`. Defaults to current term. |

### `src/modules/academics/release/actions/parent-release.action.ts`

| Action | Permission | Purpose |
|---|---|---|
| `getMyReportCardReleaseAction({ studentId, termId })` | `RESULTS_READ` | Caller is guardian of student. Returns `{ released, releasedAt?, releaseId?, isAcknowledgedByMyHousehold }`. Used by parent results page. |
| `getMyReportCardPdfUrlAction({ studentId, termId })` | `REPORT_CARDS_DOWNLOAD_OWN` | Caller is guardian. Confirms release row. Calls `_renderReportCardPdfInternal` (cache regenerates if stale). Returns 5-minute signed URL. |
| `acknowledgeReportCardAction({ releaseId, studentId })` | `REPORT_CARDS_DOWNLOAD_OWN` | Caller is guardian of student AND household is in targeted set for release. Idempotent upsert (P2002 caught → success). Audit. |

### Modifications to existing actions

**`getChildResultsAction`** in `src/modules/portal/actions/parent.action.ts`:
- After loading student + term, look up `ReportCardRelease` for `(termId, student.activeEnrollment.classArmId)`.
- If no release row, return `{ data: { released: false } }` — client renders the empty state.
- If released, hydrate `released: true`, `releaseId`, `isAcknowledgedByMyHousehold` alongside the existing payload.

**`publishResultsAction`** in `src/modules/academics/actions/result.action.ts`:
- Replace no-op body with a thin wrapper around `releaseReportCardsAction`.
- Existing `RESULTS_PUBLISH` permission preserved.
- `@deprecated Use releaseReportCardsAction directly` JSDoc.

**`renderReportCardPdfAction`** in `src/modules/academics/actions/report-card.action.ts`:
- Extract `_renderReportCardPdfInternal({ studentId, termId, schoolId })` containing the cache-check + render + upload logic.
- Public `renderReportCardPdfAction` becomes a thin wrapper that gates on `REPORT_CARDS_GENERATE` then calls the internal helper.
- The new parent-scoped action calls the internal helper directly after its own eligibility check.

**Score / remark mutations:** `mark.action.ts` and `result.action.ts`. Add cache invalidation:
- `saveSubjectMarksAction` → derive affected `(studentId, termId)` pairs, invalidate each
- `updateSubjectMarkAction` → invalidate the loaded mark's pair
- `deleteSubjectMarkAction` → same
- `recomputeTerminalResultAction` → invalidate at end of recompute

Each call wrapped in `try/catch` with `console.warn` so cache failure doesn't roll back the score edit.

### Audit + read marking

- Mutating actions (`releaseReportCards`, `reReleaseReportCards`, `chaseRelease`, `acknowledgeReportCard`) carry `audit()`.
- Read actions (`getMyReportCardReleaseAction`, `getMyReportCardPdfUrlAction`, `getReleaseStatsAction`, `getReleaseDetailsAction`, `getReleaseQueueAction`) carry `@no-audit` JSDoc.

### Error conventions

`{ data } | { error }` or `{ success } | { error }`. Key errors:
- `{ error: "Already released..." }`
- `{ error: "Report card not found" }` (generic for non-guardian / non-released)
- `{ error: "Report card not yet released" }` (PDF download path)
- `{ error: "Reminder cooldown: X hours remaining." }`
- `{ error: "All households have acknowledged. No one to remind." }`

## 8. Notifications

Two new event keys in `src/lib/notifications/events.ts`:

```ts
  REPORT_CARD_RELEASED:       "report_card_released",
  REPORT_CARD_REMINDER_SENT:  "report_card_reminder_sent",
```

**`EVENT_CHANNELS` defaults:**

| Event | Defaults |
|---|---|
| `REPORT_CARD_RELEASED` | `[in_app, email, sms]` |
| `REPORT_CARD_REMINDER_SENT` | `[in_app, email, sms]` |

Louder than D1 (which used in_app + email for circular publish, in_app + email + sms for chase) — Ghanaian schools traditionally announce results via SMS as the primary channel. Existing `RESULT_PUBLISHED` event remains in registry untouched (currently dead; not reused — different semantic).

### `src/modules/academics/release/release-notifications.ts`

```ts
export async function notifyReportCardReleased(params: {
  releaseId: string;
  termName: string;
  classArmName: string;
  studentNamesByUserId: Map<string, string[]>;
  recipientUserIds: string[];
  isReRelease: boolean;
}): Promise<void>;

export async function notifyReportCardReminder(params: {
  releaseId: string;
  termName: string;
  classArmName: string;
  studentNamesByUserId: Map<string, string[]>;
  recipientUserIds: string[];
}): Promise<void>;
```

Both honor `NotificationPreference`, fall back to defaults, swallow per-recipient errors with `console.error`. Pattern mirrors `circular-notifications.ts` from D1; module-local copy of the `fanOut` helper (no premature DRY across modules).

The `studentNamesByUserId` parameter handles multi-sibling households — one guardian may have twins in the same arm. Body interpolates the children's names: "Kofi & Akua's Term 1 results are now available."

Re-release prefixes the body: "Updated: Kofi & Akua's Term 1 report cards have been re-released..."

### Recipient resolution

Inside the calling action:
- **`releaseReportCardsAction` / `reReleaseReportCardsAction`**: load all students with active enrollment in the arm-term → join through `StudentGuardian` → `Guardian.householdId` + `Guardian.userId` → group student names per userId → dedupe userIds.
- **`chaseReleaseAction`**: same lookup, then exclude households whose all-children for this release are already acknowledged. A household is "pending" if at least one of its students has no `ReportCardAcknowledgement` row for the release.

### No post-acknowledge notification

Admin tracker reads fresh stats on page load. Matches D1.

## 10. Cache Invalidation + Edit Wiring

The existing `ReportCardPdfCache` works correctly but the invalidation pipeline is incomplete. Wire all score/remark mutations to invalidate.

**Current state:**
| Mutation | Invalidates? |
|---|---|
| `updateTerminalResultRemarksAction` | ✓ |
| `recomputeTerminalResultAction` | ✗ |
| `saveSubjectMarksAction` | ✗ |
| `updateSubjectMarkAction` | ✗ |
| `deleteSubjectMarkAction` | ✗ |

**Required wiring:** every mutation that affects a report card calls `invalidateReportCardCacheAction({ studentId, termId })`. Wrap each in `try/catch` so cache failure doesn't roll back the score edit:

```ts
try {
  await invalidateReportCardCacheAction({ studentId, termId });
} catch (err) {
  console.warn("invalidateReportCardCache failed", { studentId, termId, err });
}
```

The parent download path (`getMyReportCardPdfUrlAction`) calls `_renderReportCardPdfInternal` which uses the existing `isReportCardCacheFresh` check — stale caches regenerate on the next download. Parents always see the latest scores.

**Race conditions:**
- Edit during parent download: teacher's invalidate fires after parent's render completes. Parent gets the (now-stale) PDF; next download regenerates fresh. Eventually consistent.
- Concurrent invalidations on different students: per-student writes don't conflict.

## 11. Error Handling & Edge Cases

| Scenario | Behavior |
|---|---|
| Releasing already-released arm-term | `{ error: "Already released. Use re-release if you need to refresh." }` |
| Releasing with 0 students having results | Allowed; release row created. Parents see friendly empty state. |
| Releasing with partial coverage (e.g. 18/22) | Allowed after admin confirmation. Excluded students gain access automatically when their results land. Their households enter targeted set on next stats query. |
| Mid-term student transfer **into** the arm after release | New student's results computable; release row already grants access. They are NOT in initial notification list (already fired); admin can chase to notify. |
| Mid-term student transfer **out** of the arm after release | Release row stays. Student no longer in active-enrollment query → drops from targeted set. Existing acks preserved (audit trail). |
| Acknowledge on non-released arm | `{ error: "Report card not found" }` (generic) |
| Acknowledge for a non-guardian student | `{ error: "Report card not found" }` |
| Double-tap acknowledge | Idempotent via unique constraint (P2002 → `{ success: true }`); single audit |
| Two parents in same household acknowledge same student | Unique key `(releaseId, studentId, householdId)` → second is idempotent. `acknowledgedByUserId` records who clicked first. |
| Admin chases with pending=0 | `{ error: "All households have acknowledged. No one to remind." }` |
| Admin chases within 24h | `{ error: "Reminder cooldown: X hours remaining." }` |
| Re-release with reset on a release with acks | All ack rows for release deleted in tx. Audit logs reset. Parents see "pending" again. |
| Re-release without reset | `releasedAt` updates, `lastReminderSentAt` clears, parents re-notified ("updated"); existing acks preserved |
| Score edit after release without re-release | Cache invalidated. Next parent download regenerates. No notification. |
| Cache invalidation failure | `try/catch` in every call site. Score edit succeeds; cache stays stale until next mutation or explicit re-render. Logged. |
| Parent downloads PDF for released term, non-guardian | `{ error: "Report card not found" }` |
| Parent downloads PDF for unreleased term | `{ error: "Report card not yet released" }` |
| PDF render fails (R2 / layout) | `_renderReportCardPdfInternal` returns its existing error. Parent UI shows "Couldn't generate the PDF. Please try again later." |
| Notification per-recipient throw | Swallowed via fan-out pattern. Other recipients unaffected. |
| Multi-sibling household | Independent ack rows per child. Notification body interpolates all names. Tracker counts each student row separately. |
| Tenant isolation | Every query scoped by `schoolId`. |
| Term or class arm hard-deleted | `onDelete: Cascade` on `termId` / `classArmId` of `ReportCardRelease`; ack rows cascade through. Audit log survives. |
| Guardian deleted (GDPR) | `acknowledgedByUserId` set to null via `SetNull`; tracker shows "(deleted user)". |
| Audit write failure | Best-effort 3-retry from existing helper; action continues. |
